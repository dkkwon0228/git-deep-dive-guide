1. 개요

1.1 Git이란?

Git은 분산 버전 관리 시스템(Distributed Version Control System)입니다. 프로젝트의 변경 이력을 추적하고, 여러 사람이 협업할 수 있도록 도와주는 도구입니다.

📖 일상 비유: 작문 노트
초등학생의 일기장:

9월 1일: "오늘 학교에 갔다" (첫 번째 버전)
9월 2일: "오늘 친구와 놀았다" (두 번째 버전)
9월 3일: "오늘 숙제를 했다" (세 번째 버전)

→ 과거 일기를 지우지 않고 계속 추가
→ 언제든 과거 일기를 다시 읽을 수 있음
→ Git도 마찬가지!

1.2 왜 Git을 깊이 이해해야 하는가?

많은 개발자들이 Git 명령어만 외워서 사용합니다. 하지만 내부 동작 원리를 이해하면:

  • 문제 발생 시 스스로 해결 가능
  • 복잡한 브랜치 전략 이해
  • 효율적인 협업 가능
  • Git의 강력한 기능 100% 활용
🚗 일상 비유: 운전

명령어만 외우기

내비게이션만 따라가기
→ 길을 잃으면 당황

원리 이해하기

지도를 읽을 수 있기
→ 새로운 길도 스스로 찾을 수 있음

2. 핵심 용어 정의

2.1 Repository (저장소)

정의: Git이 프로젝트의 모든 버전을 저장하는 데이터베이스

📚 일상 비유: 도서관
도서관 = Repository
- 모든 책(파일)의 모든 판본 보관
- 초판, 재판, 3판... 모두 보관
- 원하는 판본을 언제든 꺼내 볼 수 있음

물리적 위치: .git/ 디렉토리

2.2 Working Directory (작업 디렉토리)

정의: Git으로 관리하는 파일들이 있는 최상위 폴더. 실제로 파일을 편집하는 공간.

동의어: Working Tree, Working Copy, 작업 폴더, 작업 공간

💼 일상 비유: 책상
도서관(Repository)에서 책을 빌려와서
책상(Working Directory)에 펼쳐놓고 읽고 쓰기

- 책상 = 실제 작업하는 공간
- 도서관 = 모든 버전 보관하는 곳

2.3 Staging Area (Index)

정의: 다음 커밋에 포함될 변경사항을 준비하는 중간 영역

🛒 일상 비유: 쇼핑 카트
쇼핑:
1. 매장(Working Directory)에서 물건 선택
2. 카트(Staging Area)에 담기 ← git add
3. 계산대에서 결제(Commit) ← git commit

카트에 담은 것만 구매!
마음에 안 들면 카트에서 빼기 가능!

2.4 HEAD

정의: 현재 체크아웃된 커밋(또는 브랜치)을 가리키는 포인터. "지금 어디에 있는가?"를 나타냄.

👉 일상 비유: 손가락으로 가리키기
책(커밋 히스토리):
1장 ← 2장 ← 3장 ← 4장
                   ↑
               손가락 (HEAD)
               "지금 4장을 읽는 중!"

HEAD = "현재 읽고 있는 페이지"를 가리키는 손가락

2.5 Branch (브랜치)

정의: 커밋을 가리키는 가벼운 포인터. 단순한 41바이트 텍스트 파일.

🔖 일상 비유: 책갈피
책(커밋 히스토리):
1장 ← 2장 ← 3장 ← 4장
           ↑       ↑
        책갈피A  책갈피B
        (dev)   (main)

브랜치 = 책갈피
- 같은 페이지에 여러 책갈피 가능
- 책갈피를 옮기기 쉬움
- 책갈피는 가벼움 (스티커 한 장)

2.6 Commit (커밋)

정의: 특정 시점의 프로젝트 전체 스냅샷. 변경사항을 저장소에 기록하는 것.

🎮 일상 비유: 게임 세이브
게임 플레이:
- 레벨 1 클리어 → 저장 (커밋 A)
- 레벨 2 클리어 → 저장 (커밋 B)
- 보스 직전 → 저장 (커밋 C)

커밋 = 게임 저장
- 현재 상태를 완벽히 기록
- 언제든 그 시점으로 돌아갈 수 있음

2.7 Remote (원격 저장소)

정의: 네트워크 상의 다른 위치에 있는 Repository. 보통 GitHub, GitLab 등.

☁️ 일상 비유: 구글 드라이브
내 컴퓨터 = Local Repository
구글 드라이브 = Remote Repository

- 내 컴퓨터에서 작업
- 구글 드라이브에 백업
- 다른 컴퓨터에서도 접근 가능

2.8 origin

정의: 원격 저장소의 기본 별칭(alias). URL의 짧은 이름.

📞 일상 비유: 전화번호부
전화번호부:
"엄마" → 010-1234-5678
"친구" → 010-9876-5432

Git:
"origin" → https://github.com/user/repo.git
"upstream" → https://github.com/original/repo.git

origin = 단순히 URL의 짧은 별명!

Git은 3개의 주요 영역으로 구성됩니다:

Git의 3가지 영역 흐름도

실제 파일 편집] SA[Staging Area
스테이징 영역
커밋 준비] REPO[Repository

✍️ 일상 비유: 작문 수업
1. 초안 작성 (Working Directory)
   - 공책에 자유롭게 쓰기
   - 지우개로 지우고 다시 쓰기

2. 선생님께 보여줄 부분 선택 (Staging Area)
   - 투명 필름에 제출할 부분만 복사
   - "이 문장은 빼고, 이 문장은 넣고..."
   - git add = 투명 필름에 복사하기

3. 제출 (Repository)
   - 선생님께 제출 후 점수 받음
   - 제출한 작문은 변경 불가 (영구 보존)
   - git commit = 제출하기

데이터 흐름

작업 흐름:

편집 → add → commit → push
 ↓      ↓      ↓       ↓
WD → Stage → Repo → Remote

복원 흐름:

pull → checkout → 편집
 ↓       ↓        ↓
Remote → WD ← Repo

4. 기본 개념

4.1 커밋 (Commit)

커밋이란?

정의: 현재 작업을 저장소에 기록하는 것. 프로젝트의 "스냅샷"을 찍는 것.

📸 일상 비유: 사진 앨범
여행 사진:
- 9월 1일 공항 (커밋 A)
- 9월 2일 호텔 (커밋 B)
- 9월 3일 관광지 (커밋 C)

각 사진 = 그 순간의 완전한 모습
과거 사진을 지우지 않음
언제든 과거 사진 다시 볼 수 있음

스냅샷 vs 델타

Git은 전체 스냅샷을 저장합니다! (델타 아님)

❌ 잘못된 이해 (델타)

커밋 A: 전체 파일
커밋 B: A로부터 변경사항만
커밋 C: B로부터 변경사항만

✅ 올바른 이해 (스냅샷)

커밋 A: 전체 파일 상태
커밋 B: 전체 파일 상태
커밋 C: 전체 파일 상태

→ 각 커밋은 독립적!

4.2 체크아웃 (Checkout)

체크아웃이란?

정의: 특정 커밋을 작업 디렉토리에 불러오는 것. "타임머신" 타기.

⏰ 일상 비유: 게임 불러오기
게임 저장 파일:
- Save 1: 레벨 1 클리어
- Save 2: 레벨 2 클리어
- Save 3: 보스 직전

git checkout = 저장 파일 불러오기
→ 선택한 세이브 시점으로 게임 상태가 완전히 바뀜!
커밋 vs 체크아웃
graph LR WD1[작업 디렉토리
현재 작업] REPO1[Repository
저장소] WD1 -->|commit
저장| REPO1 REPO1 -->|checkout
불러오기| WD1 style WD1 fill:#e8f4f8 style REPO1 fill:#e8f8e8

5. Git 내부 구조

5.1 .git 디렉토리 구조

.git/
├── HEAD                    # 현재 위치 포인터
├── config                  # 설정 파일
├── index                   # Staging Area
├── objects/                # 객체 데이터베이스
│   ├── 10/
│   │   └── 6287c4...      # Blob/Tree/Commit 객체
│   └── ...
├── refs/                   # 참조 (브랜치, 태그)
│   ├── heads/             # 로컬 브랜치
│   │   ├── main
│   │   └── develop
│   └── remotes/           # 원격 추적 브랜치
│       └── origin/
│           └── main
└── logs/                  # 변경 이력 로그

5.2 HEAD 파일

단순한 텍스트 파일

$ cat .git/HEAD
ref: refs/heads/main

HEAD는:

  • ✅ 일반 텍스트 파일 (41바이트)
  • ✅ cat 명령으로 볼 수 있음
  • ❌ 바이너리 파일 아님
  • ❌ 심볼릭 링크 아님

Symbolic Reference (심볼릭 참조)

HEAD는 보통 브랜치를 참조합니다:

HEAD → Branch → Commit 참조 체인
graph LR HEAD[HEAD
파일] BRANCH[main
브랜치] COMMIT[18a602e
커밋] HEAD -->|ref: refs/heads/main| BRANCH BRANCH -->|18a602e...| COMMIT style HEAD fill:#ffd700 style BRANCH fill:#87ceeb style COMMIT fill:#90ee90

Attached vs Detached HEAD

✅ Attached HEAD (정상)

$ cat .git/HEAD
ref: refs/heads/main

HEAD → main → 커밋 C
(브랜치가 자동으로 이동)

⚠️ Detached HEAD (분리)

$ cat .git/HEAD
18a602e70995b8335...

HEAD → 커밋 C
(브랜치 없음!)
⛵ 일상 비유: 배 타기
Attached HEAD = 부두에 묶인 배
  → 안전, 잃어버리지 않음

Detached HEAD = 떠있는 배
  → 위험, 떠내려갈 수 있음

6.1 Commit 객체

메타데이터만 저장

$ git cat-file -p 18a602e

tree 6335ae30...
parent 7276892...
author dkkwon0228 <email> 1761835874 +0900
committer dkkwon0228 <email> 1761835874 +0900

docs(guide): Add claude-powerline statusline setup guide

포함하는 것:

  • ✅ Tree 해시 (40바이트)
  • ✅ Parent 해시 (40바이트)
  • ✅ Author/Committer 정보
  • ✅ 커밋 메시지

포함하지 않는 것:

  • ❌ 파일 목록
  • ❌ 파일 내용
  • ❌ 디렉토리 구조
🏷️ 일상 비유: 목차 카드
도서관 카드:
- 책 제목: "Git 가이드"
- 저자: dkkwon0228
- 출판일: 2025-10-31
- 서가 위치: #6335ae30 ← Tree 해시
- 이전 판: #7276892 ← Parent 해시

카드 = Commit 객체
실제 책 내용은 서가에!

6.2 Tree 객체

디렉토리 구조 표현

$ git cat-file -p 6335ae30

040000 tree 14146669...  .claude
100644 blob 22909b0c...  .gitignore
100644 blob d45974b7...  CLAUDE.md
040000 tree b14437a9...  dashboard
040000 tree a4a7e10c...  docs
100644 blob 96398e2e...  TODO.md

형식:

  • 040000 = 디렉토리 권한
  • 100644 = 일반 파일 권한
  • tree = 디렉토리
  • blob = 파일
🗺️ 일상 비유: 서가 배치도
도서관 배치도:

1층 (Root Tree):
  - 소설 코너 → 2층 A구역 (Tree)
  - 과학 코너 → 2층 B구역 (Tree)
  - 잡지 #12345 (Blob)
  - 신문 #67890 (Blob)

배치도 = Tree 객체
실제 책 = Blob 객체

6.3 Blob 객체

실제 파일 내용 저장

특징:

  • 파일 내용 전체를 저장
  • 파일명 정보 없음 (Tree가 저장)
  • 경로 정보 없음 (Tree가 저장)
  • 순수하게 내용만!
📖 일상 비유: 실제 책
Blob = 책 내용 그 자체

책 제목? → Tree가 알려줌
어느 서가? → Tree가 알려줌
누가 썼나? → Commit이 알려줌

Blob은 순수하게 내용만 담음!

6.4 중복 제거 메커니즘

커밋 A:
  Tree A
    └─ README.md (해시: abc123)

커밋 B (README.md 변경 없음):
  Tree B (다른 Tree, 하지만...)
    └─ README.md (해시: abc123) ← 같은 해시!

objects/ab/c123 파일은 하나만 존재!
→ 두 커밋이 같은 Blob 공유
→ 저장 공간 절약!
🔧 일상 비유: 공용 부품
자동차 A:
  - 엔진 #12345
  - 타이어 #67890

자동차 B:
  - 엔진 #12345 ← 같은 부품!
  - 타이어 #11111

엔진 #12345는 한 개만 생산!
두 차가 공유 사용!

7. 브랜치 시스템

7.1 브랜치란?

정의: 커밋을 가리키는 41바이트 텍스트 파일. 가벼운 포인터.

📌 일상 비유: 포스트잇
책 (커밋 히스토리):
1장 ← 2장 ← 3장 ← 4장
              ↑      ↑
           노란색  분홍색
           (dev)  (main)

포스트잇 = 브랜치
- 같은 페이지에 여러 개 붙이기 가능
- 옮기기 쉬움
- 가벼움 (종이 한 장)
- 책을 복사하지 않음!

브랜치 파일 내용

$ cat .git/refs/heads/main
18a602e70995b83350378b863cfe557fa5141fdc

$ ls -lh .git/refs/heads/main
41바이트

# 단 41바이트!
# - SHA-1 해시: 40자
# - 줄바꿈: 1자

7.2 브랜치 생성 = 파일 하나 생성

$ git branch feature

# 내부 동작:
echo "18a602e..." > .git/refs/heads/feature

# 끝!
브랜치 생성 과정
graph LR A[Commit A] B[Commit B] C[Commit C] A --> B --> C MAIN[main
41바이트] FEATURE[feature
41바이트] MAIN -.->|18a602e| C FEATURE -.->|18a602e| C style C fill:#90ee90 style MAIN fill:#87ceeb style FEATURE fill:#ffd700

중요: 커밋을 복사하지 않음! 단지 같은 커밋을 가리키는 포인터!

🔖 일상 비유: 책갈피 추가
책의 4장에:
- 노란 책갈피 (main)
- 분홍 책갈피 (feature)

책을 복사하지 않음!
단지 책갈피 하나 더 끼움!

7.3 분기 발생

$ git checkout feature
$ git commit -m "New feature"
→ 커밋 D 생성
브랜치 분기
graph LR A[Commit A] B[Commit B] C[Commit C] D[Commit D
feature] A --> B --> C C --> D MAIN[main] FEATURE[feature] MAIN -.-> C FEATURE -.-> D style C fill:#90ee90 style D fill:#ffd700

8. 단방향 참조 구조

8.1 자식은 부모를 알지만, 부모는 자식을 모름

Git의 단방향 참조
graph RL C[Commit C
"내 부모는 B"] B[Commit B
"내 부모는 A"] A[Commit A
"부모 없음"] C -->|parent| B B -->|parent| A C -.->|"자식?"| X1[❓] B -.->|"자식?"| X2[❓] style C fill:#90ee90 style B fill:#87ceeb style A fill:#ffd700
✉️ 일상 비유: 편지 체인
할아버지 편지 (1950년):
  "친애하는 후손에게..."
  부모: (없음, 첫 편지)

아버지 편지 (1980년):
  "할아버지 편지(1950)를 읽고..."
  부모: 할아버지 편지

나의 편지 (2025년):
  "아버지 편지(1980)를 읽고..."
  부모: 아버지 편지

각 편지는:
- 자기 부모 편지는 언급 ✅
- 미래 편지는 예측 불가 ❌
- 과거는 변경 불가 ✅

8.2 왜 이렇게 설계되었나?

1. 불변성 (Immutability)

❌ 자식 목록 방식

커밋 B (자식 목록: C)

새 커밋 D 생성:
→ B를 수정해야 함
→ B의 해시 변경!
→ 전체 체인 재계산

과거가 계속 변경됨!

✅ 현재 설계

커밋 D 생성:
→ D만 작성
→ B는 변경 없음

과거는 절대 변경 안 됨!
📦 일상 비유: 타임캡슐
잘못된 설계 = 타임캡슐을 계속 열어서 내용 수정
  → "미래에 이런 일이 있었어요" 추가
  → 타임캡슐의 의미 없음

올바른 설계 = 타임캡슐은 한 번 묻으면 영원히 불변
  → 과거는 과거 그대로 보존

2. 분산 시스템의 효율성

시나리오:
당신: 커밋 B에서 커밋 C 생성
동료: 커밋 B에서 커밋 D 생성

현재 설계:
- C: parent = B
- D: parent = B
- B는 변경 없음
✅ 완벽하게 병합 가능!

9. 스냅샷 저장 방식

9.1 전체 스냅샷 vs 델타

Git은 전체 스냅샷을 저장합니다!

델타 방식 (SVN 등)

커밋 A: 전체 파일
커밋 B: A로부터 변경사항
커밋 C: B로부터 변경사항

복원 시 모든 변경사항 재생

Git 방식

커밋 A: 전체 파일 상태
커밋 B: 전체 파일 상태
커밋 C: 전체 파일 상태

각 커밋이 독립적!
📸 일상 비유: 사진 앨범
델타 방식:
- 첫 사진: 전체 모습
- 두 번째: "안경 추가"
- 세 번째: "머리 염색"
→ 복원하려면 모든 변경사항 재생 필요

스냅샷 방식:
- 첫 사진: 전체 모습 A
- 두 번째: 전체 모습 B
- 세 번째: 전체 모습 C
→ 각 사진이 독립적, 바로 볼 수 있음

9.2 중복 제거로 효율성 확보

프로젝트: 577개 파일

커밋 A: 577개 파일 전체 저장
  → 10MB

커밋 B: 2개 파일만 변경
  → 새로 저장: 2개 파일 (50KB)
  → 재사용: 575개 파일 (0바이트!)

총 증가: 50KB만!
🧱 일상 비유: 레고 블록
스냅샷 방식 = "전체 모델을 매번 조립"
  → 하지만 같은 블록은 재사용!
  → 변경된 블록만 새로 추가

레고 창고 (objects/):
- 빨간 블록 #123 (한 개만 존재)
- 파란 블록 #456 (한 개만 존재)

모델 A: 빨간 #123 + 파란 #456
모델 B: 빨간 #123 + 초록 #789
→ 빨간 블록 재사용! 초록만 추가!

10. 원격 저장소

10.1 origin의 정의

정의: 원격 저장소의 기본 별칭. URL의 짧은 이름.

$ git remote -v
origin  https://github.com/user/repo.git (fetch)
origin  https://github.com/user/repo.git (push)

10.2 fetch vs pull

✅ fetch (안전)

$ git fetch origin

동작:
1. 커밋 다운로드
2. origin/main 업데이트
3. 로컬 main 변경 안 됨!
4. 작업 디렉토리 변경 안 됨!

→ 안전! 확인만

⚠️ pull (주의)

$ git pull origin main

동작:
1. fetch 수행
2. 자동 병합!
3. 작업 디렉토리 변경됨!

→ 위험! 충돌 가능
📚 일상 비유: 도서관
fetch = 책 빌려오기 (아직 안 읽음)
- 책을 집에 가져옴
- 책장에 꽂아둠
- 현재 읽던 책은 그대로
→ 안전!

pull = 책 빌려와서 바로 읽기
- 책을 가져옴
- 즉시 읽기 시작
- 현재 내용과 섞임
→ 주의!

10.3 "브랜치가 origin/main보다 2개 커밋 앞섬"

로컬과 원격의 차이
graph LR A[A] B[B] C[C] D[D] E[E] A --> B --> C --> D --> E ORIGIN[origin/main
원격] LOCAL[main
로컬] ORIGIN -.-> C LOCAL -.-> E style C fill:#87ceeb style E fill:#90ee90

→ D, E는 아직 push 안 함!

11. 해시와 저장 메커니즘

11.1 SHA-1 해시

정의: Git이 사용하는 해시 함수. 40자리 16진수.

SHA-1("Hello Git!")
= 106287c47fd25ad9a0874670a0d5c6eacf1bfe4e
👆 일상 비유: 지문
사람마다 고유한 지문

Git:
파일마다 고유한 해시

- 같은 내용 = 같은 지문
- 다른 내용 = 다른 지문
- 지문만으로 사람 식별
- 해시만으로 파일 식별

11.2 "해시만 저장" vs "내용을 해시로 이름 붙여 저장"

❌ 잘못된 이해

해시만 저장:

.git/objects/에
해시 목록만 저장

파일 내용은?
→ 복원 불가능!

✅ 올바른 이해

내용을 저장:

.git/objects/10/6287c4...
내용: "Hello Git!"

해시 = 파일명
내용 = 파일 안에 저장!
📚 일상 비유: 도서관 보관
잘못된 이해:
서가에 책 번호만 적힌 카드
실제 책은 없음
→ 읽을 수 없음!

올바른 이해:
서가에 책이 있음
책 등에 번호표 부착
→ 번호로 찾아서 읽을 수 있음!

14. 자주 하는 오해들

❌ 오해 1: "브랜치 생성 = 커밋 복사"

진실: 브랜치 = 41바이트 텍스트 파일. 커밋 복사 없음! 단지 같은 커밋을 가리키는 포인터!

❌ 오해 2: "HEAD = 최신 커밋"

진실: HEAD = 현재 체크아웃된 커밋. 과거로 체크아웃하면 HEAD도 과거 커밋을 가리킴!

❌ 오해 3: "fetch가 원격을 변경"

진실: fetch = 읽기 전용! 원격 저장소는 전혀 변경되지 않음. 로컬의 origin/main만 업데이트!

❌ 오해 4: "해시만 저장"

진실: Git은 파일 내용 전체를 저장! 해시는 파일명으로 사용. 내용이 저장되어 있기 때문에 복원 가능!

  • 협업 능력: 복잡한 브랜치 전략 이해 및 적용
  • 자신감: 두려움 없이 Git 사용
